# Moirai Protocol — v0.1.0

The contract every agent and host adapter must follow to plug into Moirai.

This is a walking-skeleton spec. It will tighten as more agents and adapters are added. Anything marked **REQUIRED** is load-bearing; anything marked **RECOMMENDED** is convention worth following unless you have a reason not to.

---

## 1. Agent repo shape

Every agent lives in its own GitHub repository. The repo's job is to declare *what the agent is and what it needs* — no workflows, no host-specific wiring, no opinionated config values.

### REQUIRED files

| File | Purpose |
|---|---|
| `AGENT.md` | Persona, trust level, output rules, what the agent is *not*. Frontmatter starts with `name:` (agent slug, must match repo name) and `description:` (one-line). |
| `PROMPT.md` | The full prompt the LLM executes. Must read its config from `$AGENT_CONFIG_PATH` (see §3). |
| `schema.json` | JSON Schema (draft 2020-12) describing the consuming repo's `config.yml` shape. |
| `defaults.yml` | Default values for every field declared in `schema.json`. Consuming repos override; unset fields fall back here. |

### RECOMMENDED files

| File | Purpose |
|---|---|
| `scopes/` | Per-category detection logic, referenced by `PROMPT.md`. |
| `CHANGELOG.md` | So consumers can pin a tag and know what changed between versions. |
| `LICENSE` | MIT recommended for consistency across the ecosystem. |
| `README.md` | Human-facing overview, link to the Pages site if there is one. |
| `docs/` | Pages site source, if applicable. |

### Rules

- The repo MUST NOT contain workflow files. Workflows belong in the consuming repo, generated by the adapter.
- The repo MUST NOT contain a literal `config.yml` with values. Defaults live in `defaults.yml`; the live config lives in the consuming repo.
- The agent MUST NOT hard-code paths, environment names, or host names in its prompt. Use config + the environment variables in §3.

## 2. Consuming repo shape

A consuming repo submodules Moirai and installs one or more agents.

### Per installed agent

| Path | Purpose |
|---|---|
| `.github/agents/<name>/config.yml` | Live config. Must conform to the agent's `schema.json`. Any field omitted falls back to the agent's `defaults.yml`. |
| `.github/workflows/<name>.yml` | Workflow file dropped by the chosen adapter. Triggered by whatever event the agent needs (cron, pull_request, etc.). |

### Submodules

| Path | Purpose |
|---|---|
| `.moirai/` | Moirai itself, pinned to a tag. |
| `.agents/<name>/` | Each installed agent, pinned to the tag specified in Moirai's `registry.yml`. Agents are siblings of `.moirai/`, not nested inside it — git forbids submodules-within-submodules. |

The exact submodule layout is the installer's call and may evolve; consuming repos should treat it as an implementation detail.

## 3. The envelope — environment variables

Every adapter MUST inject the following environment variables into the agent's execution context:

| Variable | Value | Purpose |
|---|---|---|
| `AGENT_NAME` | The agent's slug from the registry (e.g. `curator`). | Lets the agent self-identify for logging, issue titles, etc. |
| `AGENT_CONFIG_PATH` | Absolute path to the consuming repo's `.github/agents/<name>/config.yml`. | The agent reads its config from this path. |
| `AGENT_REPO_DIR` | Absolute path to the agent's own submodule directory in the consuming repo. | Lets the agent read its own `defaults.yml`, `schema.json`, `scopes/`, etc. |
| `TODAY` | ISO date (`YYYY-MM-DD`) at run time, UTC. | For dated outputs (issue footers, audit blocks, etc.). |

### Trigger-specific envelope (REQUIRED for `pull_request` triggers)

When the agent's trigger is `pull_request`, the adapter MUST additionally inject:

| Variable | Value | Purpose |
|---|---|---|
| `PR_NUMBER` | The pull request number (e.g. `42`). | Lets the agent address the PR (e.g. in issue titles, audit block headers). |
| `PR_BASE_SHA` | Base commit SHA (typically the merge target's HEAD). | For computing the diff: `git diff $PR_BASE_SHA $PR_HEAD_SHA`. |
| `PR_HEAD_SHA` | Head commit SHA (the PR tip). | The other side of the diff. |

These MUST also be resolvable when the same workflow is invoked via `workflow_dispatch` with a `pr_number` input (manual-run mode). The standard adapter implementation resolves them via the GitHub event context for `pull_request` and via `gh pr view` for `workflow_dispatch`.

Adapters MAY inject additional host-specific variables. Agents that need them MUST document them in their `AGENT.md`.

## 4. Sinks

Every agent declares which sink(s) it writes to. The sink is the destination for the agent's output and shapes how its findings are surfaced and consumed.

### Defined sinks

| Sink | Description | Example |
|---|---|---|
| `issues` | Filed as GitHub Issues with stable title prefix + labels. Best for durable backlog. | Curator |
| `inbox` | Appended as a dated block to a markdown file in the consuming repo (default `notes/inbox.md`). Best for session-scoped findings surfaced by `pickup`-style skills. | Auditor |
| `summary` | Written to the workflow's step summary. Ephemeral, run-bound. Useful for dry-run modes. | Curator's dry-run |
| `pr-comment` | Posted as a sticky comment on the triggering pull request. | Reserved for future PR-feedback agents |

### Sink rules

- The agent's `defaults.yml` MAY declare a default sink. The consuming repo's `config.yml` MAY override it.
- When writing to `inbox`, the agent MUST use a stable, machine-readable header (e.g. `## Audit — YYYY-MM-DD (PR #N)`) so re-runs can replace the prior block idempotently.
- When writing to `issues`, the agent MUST use a stable title prefix (typically `[<agent-name>] <category>:`) and dedup against existing open issues before filing.
- Agents reading other agents' output MUST do so via these stable schemas, not via free-text scraping.

### `inbox` sink commit-back

When the sink is `inbox`, the adapter is responsible for committing the agent's edit to the inbox file back to the source branch. The agent only writes the file — it does NOT call `git commit` or `git push` itself.

The destination of the commit-back depends on the trigger:

| Trigger | Push target | Commit message shape |
|---|---|---|
| `pull_request` | PR head ref (from the resolved PR context) | `chore(<agent>): refresh PR #N findings` |
| `schedule` / `workflow_dispatch` | The workflow's own ref (`${{ github.ref_name }}`, typically `main`) | `chore(<agent>): nightly findings — YYYY-MM-DD` |

The standard `claude-code-action` adapter handles both. Both:
- Diff the inbox file after the agent runs; exit cleanly if unchanged.
- Configure git as `<agent-name>-bot` (e.g. `auditor-bot`).
- Commit only the configured inbox path.

Adapters MUST NOT commit anything else.

### Fork safety

For `pull_request` triggers with the `inbox` sink, the adapter MUST skip when the PR head is on a fork. Push permissions for `auditor-bot` and similar do not extend to fork branches; attempting the commit-back would fail loudly. The skip step should `exit 0` with a `::notice::` line, not fail the workflow.

Fork safety does not apply to `schedule` triggers because the commit-back goes to the workflow's own ref (typically `main`), not to any PR's head. Agents in schedule mode MAY audit fork-PRs by reading their diffs via `gh pr view`; the resulting inbox blocks land in the consuming repo's own branch.

## 5. Adapter responsibilities

An adapter is the bridge between a host environment (today: GitHub Actions + Claude Code) and the protocol. The installer generates a workflow file from the chosen adapter's template.

An adapter MUST:

1. **Load the prompt** — read the agent's `PROMPT.md` from `$AGENT_REPO_DIR` and pass it to the LLM.
2. **Inject the envelope** — set every variable in §3 in the LLM's execution environment.
3. **Authenticate** — handle the LLM provider's auth (e.g. `anthropic_api_key`) and the host's auth (e.g. `github_token`) according to the host's conventions.
4. **Provide tool access** — grant the agent the tools its `AGENT.md` declares it needs (typically Bash, Read, plus sink-specific tools like `mcp__github__create_issue` for the `issues` sink).
5. **Respect kill switches** — check for any pause file the agent declares before invoking the LLM (e.g. `.<agent>-pause` at repo root).

An adapter MAY provide additional features (sticky-comment dedup, idempotent run management) but MUST NOT add behaviour the agent didn't request.

---

## 6. Versioning

- Agents follow [SemVer](https://semver.org/). Breaking changes to the prompt, scopes, or config schema bump the major version.
- Moirai's `registry.yml` pins each agent to a specific tag. Consuming repos get the pinned version when they install.
- Bumping an agent's pinned version in `registry.yml` is a Moirai-side change, surfaced via PR.

---

## 7. Out of scope (for now)

- **Cross-LLM portability.** Today, Moirai assumes a Claude Code adapter. Future adapters for OpenAI, local CLI, etc. are anticipated but not specified yet.
- **Inter-agent dispatch.** The protocol enables agents to read each other's sinks but does not yet define a coordinator agent. That comes after step 4 of the walking skeleton.
- **Hosted runtime.** Moirai is CI-driven for now. A hosted execution environment is not in scope.
